## Critical

### C1 — Paid-unlock columns were client-writable ✅ `2f8f852`
`cases.dispute_paid` / `dispute_unlock_source` / `promo_code` had table-level
UPDATE/INSERT granted to `authenticated` and `anon`. RLS gates the *row* but not
*columns*, so a signed-in user could PATCH their own case to `dispute_paid=true`
via PostgREST and unlock the paid dispute package for free.
**Fix:** revoked table-level UPDATE/INSERT from the client roles and re-granted
per-column on every column *except* the three protected ones; `service_role`
keeps full access. Legit writes already run as service role (Stripe webhook,
promo redeem). Migration `20260612130000_lock_paid_unlock_columns.sql`, applied
live and grant-verified.

### C2 — No size cap / rate limit before Anthropic calls ✅ `1ac9e75`
(Paired with **H6**.) `/api/audit-guest` (public) and `/api/extract` had no
payload-size ceiling and no throttle, so either could be driven to run up an
unbounded Anthropic bill.
**Fix:** reject decoded payloads over `MAX_FILE_BYTES` (20 MB) before any model
call (413); Postgres-backed fixed-window limiter (`rate_limits` table +
`rate_limit_hit` RPC, service-role only) — per-IP on guest (15/10 min), per-user
on extract (30/10 min), 429 on exceed, fails open. Migration
`20260612131000_rate_limits.sql`, applied live and RPC-verified.

## High

### H1 — Dead routes + unused richer extractor ✅ `d71e9c1` (swap ⏸️)
`/api/audit` and `/api/extract-line-items` had no callers.
**Fix:** both deleted. Evaluated replacing the live text-JSON extractor
(`extraction.ts`) with the tool-use extractor in `billExtractor.ts`: it is more
robust and captures richer metadata, but the swap changes extraction for every
upload, drops the `lowConfidence` signal, and needs real-upload validation — so
it is **deferred** to its own change (rationale documented in `billExtractor.ts`).
`billExtractor.ts` is retained regardless because `extraction.ts` imports its
CPT-mapping helpers.

### H2 — Signed-in upload dead-ended on a fake "we'll email you" page ✅ `3154cff`
Signed-in uploads now run the audit and redirect to the case page.

### H3 — Failed extraction stranded a case in `auditing` forever ✅ (H3/M6/M7/L1 pass)
A case is created (status `auditing`) before `/api/extract` runs. If extraction
failed, the upload flow swallowed the error and the case page showed "Audit in
progress… check back" with no way to re-run — a permanent dead end.
**Partial fix already present:** the phase-machine upload flow (`cab1c05`) now
surfaces an extract failure in-session with a Retry. **Remaining fix:** the
case page's `auditing` state now says the audit didn't finish and offers a
**Re-run audit** control that re-selects the bill and calls `/api/extract`
again, then reloads. (The original file isn't persisted, hence re-select.)

### H4 — Same bill scored differently as guest vs signed-in ✅ `8325120`
Guest ($924) and signed-in ($851) used different `potentialSavings` formulas.
Unified on one `runFullAudit` / `computeRecoverable` pipeline used by all three
audit paths; the claim path re-audits server-side instead of trusting client
numbers.

### H5 — Patient-dispute analysis only existed on the dead /api/audit ✅ `d71e9c1`
`analyzeDisputedProcedures` now runs inside `runFullAudit` as an optional,
best-effort step (gated on a patient note, non-fatal on failure); `/api/extract`
passes the case's `userNotes`, restoring patient-reported not-rendered/disputed
flagging for signed-in audits.

### H6 — Per-IP / per-user throttling ✅ `1ac9e75`
See **C2** (delivered together).

## Medium

### M1 — Webhook unlocked on completed-but-unpaid sessions ✅ `8dc3db9`
`checkout.session.completed` set `dispute_paid` on `mode === 'payment'` without
checking `payment_status`. Now requires `session.payment_status === 'paid'`.

### M2 — Webhook idempotency race ✅ `8dc3db9`
Select-then-insert-at-end let two concurrent deliveries both process. Now the
`webhook_events` row (PK `stripe_event_id`) is inserted **first**; a duplicate-key
(23505) is treated as already-processed, and the claim row is deleted on handler
failure so Stripe's retry re-runs.

### M3 — /api/outcomes had no ownership check ✅ `1c0a094`
Upsert keyed on a client-supplied `outcomeId` with no check that an existing row
belonged to the caller. Now looks the row up via the service role and returns a
clean 403 on cross-user IDs (defense in depth atop RLS).

### M4 — Dashboard "total recovered" summed the wrong field ✅ `1c0a094`
Summed `potential_savings` instead of `amount_recovered`, overstating recoveries.
Fixed to sum `amount_recovered`.

### M5 — `cases.user_id` was nullable ✅ `1c0a094`
Restored `NOT NULL` after purging leftover ownerless guest demo rows
(dispute_letters / advocacy_workflows cascade-delete). Migration
`20260612132000_cases_user_id_not_null.sql`, applied live and verified.

### M6 — `normalizeCBSSet` mutated its input documents ✅ (H3/M6/M7/L1 pass)
The normalizer pushed computed discrepancies/temporal flags onto the caller's
`doc.discrepancies` / `doc.temporalInconsistencies` and returned the same
objects, so re-running an audit on an already-normalized/persisted set
double-appended discrepancies. **Fix:** clone the documents with fresh arrays at
the top of `normalizeCBSSet` and annotate the clones — pure function, idempotent.

### M7 — Outcome predictions silently blended against empty data server-side ✅ (H3/M6/M7/L1 pass)
`outcomePrediction.ts` imported `getAggregateStats` from the `'use client'`
localStorage store; server-side `window` is undefined, so the "real outcomes"
blend was always zero. (Currently only called client-side, so latent — but the
parallel session is moving this neighborhood server-side.) **Fix:**
`predictAll`/`predictOutcome` accept optional injected `OutcomeStats`; added
`lib/predictions/serverStats.ts::aggregateStatsFromSupabase()` which reads
`dispute_outcomes` from Supabase in the same shape. Client path unchanged;
server callers pass real data. Coordinated to avoid touching the
Anthropic-in-client-bundle work in the parallel session.

## Low

### L1 — Stale "error report within 24 hours" framing ✅ (mostly pre-fixed)
Synchronous audits made the "report within 24 hours" framing wrong. **Status:**
the upload page was fixed in `3154cff`/L-items ("Full error report in minutes");
the dashboard EmptyState carries no time-delay copy. The case page's `auditing`
state still implied passive auto-population ("will populate… as soon as it's
ready") — rewritten in the H3 pass to state the audit didn't finish and offer a
re-run. The remaining "24 hours" strings live only in landing (`page.tsx`) and
`how-it-works` — those are **marketing copy (M8), intentionally untouched**.

### L2 — Two different submission deadlines for one case ✅ (L-items pass)
Letter page hardcoded "letter date + 30 days" while the case page's
`DeadlineTracker` used the per-rule calculator. New `lib/deadlines/forCase.ts`
(`cbsSetForCase` / `deadlinesForCase`) is the single source: both pages now
derive deadlines from `calculateDeadlines`. The letter banner shows the most
pressing rule-based deadline.

### L3 — "Audit another bill" left stale inputs ✅ (L-items pass)
Reset only cleared `billFile`/`step`, leaving `eobFile`/`careType`/
`insuranceType`/`gfe`/`userNotes`. Now resets all inputs.

### L4 — /api/generate-letter injected unbounded `errors`/`caseData` into the prompt ✅ (L-items pass)
Added shape validation + size caps (≤100 errors, per-field string caps, userNotes
≤4000 chars) before the Anthropic call; the prompt uses the sanitized values.

### L5 — /api/outcomes accepted client `created_at` and raw numerics ✅ (L-items pass)
Users could backdate rows. `created_at` is now the DB default (never client-set);
numeric fields are coerced/clamped; `resolved_at` and `status` are validated;
free-text is length-capped.

### L6 — Fee schedule prices at national locality only 🗺️ Roadmap (not implemented)
`mockFeeSchedule` prices PFS at the national locality (TODO: ZIP→locality), so
overcharge dollar figures are geography-blind. **Decision:** roadmap item —
documented, not implemented in this pass.

### L7 — Duplicate inline `mapInsuranceType` ✅ (L-items pass)
audit-guest / extract / claim each had a slightly different inline mapper
(default `commercial`) vs the canonical `lib/insuranceMapping.ts` (default
`other`). All three now use `normalizeInsuranceType`.

### L8 — Dashboard Digital-Twin faked error count ✅ (L-items pass)
`errorCount` was `potential_savings > 0 ? 1 : 0`, undercounting multi-error
cases. Now uses `errors_found.length` (added to the dashboard query).

## Verified safe (checked, no change needed)
From the audit's verified-safe list:
- **Payment tables are not a free-membership vector** — `subscriptions` /
  `payments` / `profiles` are SELECT-own-only with no client INSERT/UPDATE;
  `webhook_events` is service-role-only, deny-by-default.
- **Webhook signature verification is correct** — raw `request.text()` +
  `constructEvent`, `runtime = nodejs`.
- **Password gate has no bypass** — constant-time compare, hashed cookie,
  open-redirect guard.
- **No server secret in client / `NEXT_PUBLIC` code.**
- **The June 12 isolation fixes held** — `/api/cases`, `/api/em-review`,
  `/api/outcomes` require auth and scope correctly; no `using(true)` policies
  remain.

Also confirmed sound while remediating:
- **Case detail render vs stored schema** — stored `errors_found` /
  `bill_data.normalizedCbs` are well-formed and the render tree is defensive; no
  deterministic schema-driven crash was reproducible from live data.
- **`webhook_events`** has a PRIMARY KEY on `stripe_event_id` (enables M2).
- **`dispute_outcomes` RLS** — `auth.uid() = user_id` on SELECT/UPDATE/DELETE.
- **Protected-column writes** (`dispute_paid` etc.) all flow through the
  service-role admin client (Stripe webhook, promo redeem).
