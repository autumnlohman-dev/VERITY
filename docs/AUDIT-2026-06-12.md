## Critical

### C1 тАФ Paid-unlock columns were client-writable тЬЕ `2f8f852`
`cases.dispute_paid` / `dispute_unlock_source` / `promo_code` had table-level
UPDATE/INSERT granted to `authenticated` and `anon`. RLS gates the *row* but not
*columns*, so a signed-in user could PATCH their own case to `dispute_paid=true`
via PostgREST and unlock the paid dispute package for free.
**Fix:** revoked table-level UPDATE/INSERT from the client roles and re-granted
per-column on every column *except* the three protected ones; `service_role`
keeps full access. Legit writes already run as service role (Stripe webhook,
promo redeem). Migration `20260612130000_lock_paid_unlock_columns.sql`, applied
live and grant-verified.

### C2 тАФ No size cap / rate limit before Anthropic calls тЬЕ `1ac9e75`
(Paired with **H6**.) `/api/audit-guest` (public) and `/api/extract` had no
payload-size ceiling and no throttle, so either could be driven to run up an
unbounded Anthropic bill.
**Fix:** reject decoded payloads over `MAX_FILE_BYTES` (20 MB) before any model
call (413); Postgres-backed fixed-window limiter (`rate_limits` table +
`rate_limit_hit` RPC, service-role only) тАФ per-IP on guest (15/10 min), per-user
on extract (30/10 min), 429 on exceed, fails open. Migration
`20260612131000_rate_limits.sql`, applied live and RPC-verified.

## Medium

### M1 тАФ Webhook unlocked on completed-but-unpaid sessions тЬЕ `8dc3db9`
`checkout.session.completed` set `dispute_paid` on `mode === 'payment'` without
checking `payment_status`. Now requires `session.payment_status === 'paid'`.

### M2 тАФ Webhook idempotency race тЬЕ `8dc3db9`
Select-then-insert-at-end let two concurrent deliveries both process. Now the
`webhook_events` row (PK `stripe_event_id`) is inserted **first**; a duplicate-key
(23505) is treated as already-processed, and the claim row is deleted on handler
failure so Stripe's retry re-runs.

### M3 тАФ /api/outcomes had no ownership check тЬЕ `1c0a094`
Upsert keyed on a client-supplied `outcomeId` with no check that an existing row
belonged to the caller. Now looks the row up via the service role and returns a
clean 403 on cross-user IDs (defense in depth atop RLS).

### M4 тАФ Dashboard "total recovered" summed the wrong field тЬЕ `1c0a094`
Summed `potential_savings` instead of `amount_recovered`, overstating recoveries.
Fixed to sum `amount_recovered`.

### M5 тАФ `cases.user_id` was nullable тЬЕ `1c0a094`
Restored `NOT NULL` after purging leftover ownerless guest demo rows
(dispute_letters / advocacy_workflows cascade-delete). Migration
`20260612132000_cases_user_id_not_null.sql`, applied live and verified.

## Low

### L1 тАФ тЭФ TEXT NOT RECOVERED

### L2 тАФ Two different submission deadlines for one case тЬЕ (L-items pass)
Letter page hardcoded "letter date + 30 days" while the case page's
`DeadlineTracker` used the per-rule calculator. New `lib/deadlines/forCase.ts`
(`cbsSetForCase` / `deadlinesForCase`) is the single source: both pages now
derive deadlines from `calculateDeadlines`. The letter banner shows the most
pressing rule-based deadline.

### L3 тАФ "Audit another bill" left stale inputs тЬЕ (L-items pass)
Reset only cleared `billFile`/`step`, leaving `eobFile`/`careType`/
`insuranceType`/`gfe`/`userNotes`. Now resets all inputs.

### L4 тАФ /api/generate-letter injected unbounded `errors`/`caseData` into the prompt тЬЕ (L-items pass)
Added shape validation + size caps (тЙд100 errors, per-field string caps, userNotes
тЙд4000 chars) before the Anthropic call; the prompt uses the sanitized values.

### L5 тАФ /api/outcomes accepted client `created_at` and raw numerics тЬЕ (L-items pass)
Users could backdate rows. `created_at` is now the DB default (never client-set);
numeric fields are coerced/clamped; `resolved_at` and `status` are validated;
free-text is length-capped.

### L6 тАФ тЭФ TEXT NOT RECOVERED

### L7 тАФ Duplicate inline `mapInsuranceType` тЬЕ (L-items pass)
audit-guest / extract / claim each had a slightly different inline mapper
(default `commercial`) vs the canonical `lib/insuranceMapping.ts` (default
`other`). All three now use `normalizeInsuranceType`.

### L8 тАФ Dashboard Digital-Twin faked error count тЬЕ (L-items pass)
`errorCount` was `potential_savings > 0 ? 1 : 0`, undercounting multi-error
cases. Now uses `errors_found.length` (added to the dashboard query).
